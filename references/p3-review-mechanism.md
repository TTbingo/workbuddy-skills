# P3 定期回顾机制（持续进化）

> **目的**：基于真实辅导记录持续更新 gotchas.md，让 Skill 从"理论最优"进化到"实战最优"。
> **触发时机**：每次真实辅导后（用户说"辅导完了"或类似表达）
> **执行频率**：每次辅导后即时记录 + 每月定期回顾（或每 10 次辅导后）

## 执行步骤

**Step P3-1：收集反馈**
- 主动问用户："这次辅导有什么特别的地方吗？比如：孩子卡在哪了？有没有出现意外的情况？"
- 如果用户提供了反馈或描述了失败场景 → 进入 Step P3-2
- 如果用户说"没问题" → 跳过，不强制回顾

**Step P3-2：提炼失败模式**
- 分析反馈中是否有**新的失败模式**(不在 G1-G24 中的)
- 如果有 → 提炼成新 gotcha(G25, G26, ...)，按 `gotchas.md` 模板写入
- 无论是否有新 gotcha → 记录到 `references/tutor-log.md`（按模板追加）

**Step P3-3：更新 gotchas.md**
- 如果是新 gotcha → 插入到 `gotchas.md` 的对应位置（按主题分类）
- 如果是已有 gotcha 的新案例 → 在对应 gotcha 的"失败场景"中补充
- 更新 gotchas.md 开头的"状态"标记(如"已验证 G1-Gx")

**Step P3-4：检查约束有效性**
- 回顾本次辅导是否遵守了「品味编码」中的 if-then 约束
- 如果约束被执行但效果不好 → 调整约束的 if 条件或 then 动作
- 如果约束没有被触发 → 检查是否需要增加触发条件

## 输出要求

- **必做**：更新 `references/tutor-log.md`（追加本次辅导记录）
- **条件必做**：如果发现新 gotcha → 更新 `references/gotchas.md`
- **可选**：如果发现约束问题 → 更新 SKILL.md 中的对应约束

## 定期回顾触发词

用户说以下表达时，主动执行 P3 机制：
- "辅导完了"
- "这次辅导怎么样？"
- "有什么需要改进的吗？"
- "帮我总结一下这次辅导"

## 🤖 自动触发机制（方案C，2026-07-01）

> 除了用户手动触发，以下条件自动启动 P3 回顾流程，不等用户提醒。

### 触发条件矩阵

| 触发类型 | 条件 | 动作 | 优先级 |
|:---|------|------|:---:|
| **时间驱动** | 距上次 P3 回顾 ≥30 天 | 自动执行完整 P3 回顾（4步） | P1 |
| **数量驱动** | 辅导记录 ≥10 条（tutor-log.md 新增条目） | 自动执行快速回顾（Step P3-1 + P3-2） | P1 |
| **版本驱动** | 版本号大版本变动（1.x.0→2.0.0 或 x.0.0→x+1.0.0） | 自动执行完整 P3 回顾 + gotchas 健康度重扫 | P0 |
| **累积驱动** | 新增 gotcha ≥5 条（自上次回顾以来） | 触发去重归并检查 | P2 |

### 自动执行流程

```
检测到触发条件
  ↓
① 读取 references/tutor-log.md（统计新增条数）
② 读取 references/gotchas.md（检查健康度/触发计数/老化标记）
③ 执行去重归并（标题相似度 ≥80% → 归并）
④ 标记 🧟 候选（30天零触发 → 标记；60天 → 淘汰候选）
⑤ 更新 gotchas.md 健康度分析节
⑥ 输出 P3 回顾报告（简短版：新增N条/归并M对/淘汰候选K条/建议）
```

### 回顾报告格式

```
📊 P3 定期回顾（2026-0X-XX）
触发原因：[时间驱动 / 数量驱动 / 版本驱动]

📈 辅导统计：新增 X 条记录 / 累计 Y 条
🐛 Gotchas：新增 N 条 / 归并 M 对 / 🧟 候选 K 条
🔧 约束变更：需要调整的约束 0 条

⚠️ 待关注：
- [如有] GXX 连续 60 天零触发 → 建议淘汰
- [如有] 出现新失败模式但未追加 gotcha

💡 建议：...
```
